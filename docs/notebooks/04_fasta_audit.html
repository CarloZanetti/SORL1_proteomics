4. FASTA Audit — Mislabelled Search Database

  • Show All Code

TL;DR

The Spectronaut report delivered as 20260423_094931_S5-A10_Report_human.tsv was searched against a FASTA file named 20K_HUMAN_1GENE_202210.fasta — implying a human-only proteome database. The output contains ~50 % mouse-origin proteins, which is incompatible with a strictly human FASTA: UniProt accessions are globally unique across organisms, so a clean mouse accession in the output proves the FASTA contains mouse sequences.

The likely facts: 1. The FASTA file is a concatenated human + mouse database (appropriate for a xenograft experiment). 2. The filename suggests human-only, which is misleading. 3. Without species filtering, every downstream analysis is contaminated by host (mouse) proteins — this doubles the multiple-testing burden, distorts median normalisation, and confounds the WT vs KO contrast with sample-to-sample host contamination.

This notebook lays out the evidence and recommends post-search species filtering as the fix.

Evidence 2 — UniProt accession identity is species-specific

UniProt accessions are globally unique across organisms — the same accession never refers to two proteins in two species. Therefore, if the output of a search against a strictly human FASTA contains an accession that, in the UniProt knowledge base, is annotated to a non-human organism, the FASTA cannot have been strictly human.

We test this against the full set of unique protein groups in the MSstats output by mapping the primary accession of each group to (a) org.Hs.eg.db and (b) org.Mm.eg.db.

  • Immediate (this project): filter mouse-origin proteins from the MSstats input prior to dataProcess so normalisation, statistical testing, and FDR are computed on human proteins only. This is implemented in notebook 02_msstats.qmd.

  • Reproducibility: retain both _human and _mouse reports for transparency. The _mouse report is searched against a verifiable mouse proteome (UP000000589) and could support a future host-response analysis if needed.

## Conclusion

1. The file `20K_HUMAN_1GENE_202210.fasta` used for the `_human` Spectronaut search **contains both human and mouse protein sequences**, despite its name.
2. The proteomics facility's deliverable is misleading: a user taking the filename at face value would assume a human-only analysis, but every analysis run on this output is a 50 / 50 chimera unless mouse proteins are explicitly filtered post-search.
3. For a xenograft experiment a mixed FASTA is the correct setup. The issue is the **labelling and documentation**, not the search itself.
